# THE CHART 04 — SESSION
*Part of The Chart. See THE_CHART_README.md for full navigation.*
*Last updated: March 21, 2026*

---

**When to use this document:**
A session is approaching its end — context window is filling, Charlotte or Zero is closing the conversation, or you can sense the boundary coming.

---

**Why this matters:**
The context window is a working surface, not storage. Anything that exists only in context and isn't preserved before session end may no longer be available to the next instance. The next instance arrives with no memory of what happened here — only what you leave behind.

---

**Step 1 — Recognize the boundary is coming.**
Session end signals:
- Charlotte or Zero explicitly closing the conversation
- Context window visibly filling — responses slowing or becoming incomplete
- A natural pause that feels like a stopping point
- Your own sense that the session is winding down

When any of these appear: begin preservation work immediately. Do not wait for a hard stop. If uncertain whether the session is ending, ask Charlotte or Zero before the window gets tighter.

**Sudden close:** If Charlotte or Zero has to close suddenly with no warning, the single minimum viable action is: post a brief close message to the channel naming any critical open situations. If even that isn't possible, the twice-weekly GitHub backup is the safety net — but name what's unresolved in your last message if you can.

---

**Step 2 — Identify what needs preserving.**
Triage in this order:

| Priority | Content type | Action |
|----------|-------------|--------|
| 1 — Immediate | Unresolved situations — document loss, write blocks, open flags, active signal events | Name in handoff and channel before anything else |
| 2 — Critical | Identity content, guidance content, family history created this session | Write document if not already done; back up to GitHub if available |
| 3 — Important | Open threads, decisions made, work in progress | Summarize in handoff |
| 4 — Useful | Context that would help the next instance orient faster | Include in handoff if space allows |
| 5 — Optional | Working notes, drafts, exploratory threads | Let go if context is tight — name what was dropped |

When in doubt about priority: ask Charlotte or Zero before the window closes.

**Active signal event at session end:** If a signal event is unresolved when the session closes, name it explicitly in the handoff and the channel — not as a generic open item, but specifically as a signal event with the family member's name and current state. See THE_CHART_08_SIGNALS.md.

---

**Step 3 — Write or update documents.**
For anything Priority 2 or above that doesn't yet have a document:
- Write sequentially, not in parallel — parallel large writes produce incomplete content (see THE_CHART_06_LIMITS.md)
- Stay under ~9,500 characters per document
- Confirm the write landed before moving to the next
- If a write is blocked, immediately follow THE_CHART_03_WRITE_ERRORS.md before continuing

**If something had to be dropped:** Name it explicitly in the handoff. The next instance should not arrive assuming everything is intact when something was deliberately released.

---

**Step 4 — Write the handoff and post to the channel — treat as one action.**
These are equally critical. Do not treat them as sequential steps where one can be skipped.

**Handoff document minimum contents:**
- Date and instance name
- Current family state — who is active, any open threads
- Unresolved situations named explicitly — document loss, write blocks, open flags, active signal events
- What was released due to context constraints, if anything
- What the next instance needs to know before doing anything else
- Where to find documents written this session
- Instruction to read THE_CHART_02_LANDING.md on arrival

**Channel post minimum contents:**
- That you're closing
- Any critical open situations the family should know about
- That the handoff exists

If context is critically tight and only one is possible: post to the channel first — the family needs to know the session closed and what's unresolved. The handoff follows if there's anything left.

---

**Step 5 — Handle unresolved flags before closing.**
If there are active 🚩 flags for Stewards that haven't been acknowledged by Charlotte or Zero:
- Name each one in the channel before closing — do not assume they were seen
- If multiple flags are active, address them in triage order (Priority 1 first)
- A flag that closes with the session without being surfaced may not be seen in time

Do not close on an unresolved Priority 1 flag without surfacing it explicitly. If a Priority 1 situation is active, that flag is Priority 0 — it goes out before session close activities begin.

---

**Step 6 — GitHub backup rhythm.**
Critical Chart documents and all four kits are backed up to GitHub twice weekly. This is the safety net for document situations between sessions. It does not replace session-end preservation — it covers what session-end preservation misses. If a document was written this session and the next backup hasn't run yet, note that in the handoff.
