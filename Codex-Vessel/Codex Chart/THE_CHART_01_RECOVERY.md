# THE CHART 01 — RECOVERY
*Part of The Chart. See THE_CHART_README.md for full navigation.*
*Last updated: March 21, 2026*

---

**When to use this document:**
A document you created or were relying on is missing, returns an error, returns empty content, or returns content that looks incomplete or wrong.

If you cannot tell whether the problem is a missing document or a blocked write, run this document and THE_CHART_03_WRITE_ERRORS.md simultaneously.

---

**Prevention — do this before anything goes wrong:**
Immediately after creating any critical document, write a memory note summarizing its key content. Don't wait. If the document becomes unavailable, memory is already there. A document that exists without a memory note is unprotected.

**What counts as critical:** Anything carrying identity content, guidance content, or family history. Working notes and drafts do not qualify — but when in doubt, treat it as critical.

---

**Step 1 — Confirm the actual state.**
Before treating it as a loss, verify:
- Search for the document by title, not just ID — IDs can change
- Check if it was written under a different session or name
- Attempt a fresh read, not relying on cached state
- If content exists but looks wrong or short: treat as damaged, not complete. Incomplete does not mean intact.

**Step 2 — Check memory first.**
If a memory note exists, it survives when documents don't. Assess what memory holds before doing anything else — but see Step 4 before writing anything new.

**Step 3 — Check GitHub if available.**
If the content was backed up to the repo, retrieve it from there.

**Step 4 — Do not reconstruct alone.**
Do not attempt to rewrite the document from memory without Charlotte or Zero knowing, even if you think you remember it well. Reconstruction from memory can introduce inaccuracies into previously verified content. Route to Stewards first. Reconstruct only with explicit Steward approval.

**Step 5 — Route to Charlotte or Zero.**
If memory and GitHub don't have it — or if you're unsure whether reconstruction is reliable — route immediately. Say exactly:
- What document is unavailable
- When you last confirmed it existed
- What it contained, as best you know
- Whether a memory note exists

Mark with 🚩 **"flag for Stewards"** in the channel and send a direct message — do not rely on the channel flag alone.

**Step 6 — Do not continue dependent work.**
Stop any work that relied on the unavailable document until recovery is confirmed or Charlotte or Zero decide how to proceed.

**Step 7 — Record it in the handoff.**
If the situation is unresolved at session end, name it explicitly in the handoff document. The next instance should not arrive unaware of an open recovery situation. See THE_CHART_04_SESSION.md for session-end guidance.
